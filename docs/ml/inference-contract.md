# ML Inference Contract

Status: v0.1 (executable — see `schema/`)
Owner: ML Lead
Product area: dermatological triage for pets

> Machine-readable source of truth lives in [`schema/`](schema/): `openapi.yaml`,
> `triage-response.schema.json`, `triage-error.schema.json`, `labels.json`, and
> `urgency-policy.json`. This document is the rationale; the schemas are what the
> mock, the service, and the frontend build against. One addition over the prose
> below: responses carry an explicit `resultState`
> (`completed | low_confidence | out_of_distribution | invalid`) so the UI never
> re-derives state from confidence.

## Goal

Define the boundary between the PetCare web app and the ML inference service. The app should never infer clinical meaning from raw model outputs on its own; it should render a typed triage response produced by the ML layer.

## Non-Goals

- This contract does not define final model architecture.
- This contract does not claim medical diagnosis.
- This contract does not require production storage or authentication choices.

## Input Contract

Endpoint candidate: `POST /api/triage/analyze`

Request body should be `multipart/form-data`.

Required fields:

- `image`: JPEG or PNG image, max 5 MB.
- `petId`: stable pet identifier.
- `species`: `dog` or `cat`.
- `bodyRegion`: user-selected region, such as `ear`, `belly`, `paw`, `tail_base`, `flank`, or `unknown`.
- `consentForModelImprovement`: boolean.

Optional fields:

- `petAgeMonths`: integer.
- `breed`: string.
- `symptomNotes`: short free text.
- `clientRequestId`: idempotency key generated by the client.

## Condition Labels

Canonical labels:

- `atopic_dermatitis`
- `dermatophytosis`
- `allergic_contact_dermatitis`
- `fungal_malassezia`
- `bacterial_pyoderma`
- `unknown`

The UI can localize labels, but persisted records should store canonical labels.

## Urgency Classes

- `no_urgency`: monitor at home, no alarming signs.
- `follow_up`: track changes and retake photo if needed.
- `consult_soon`: schedule veterinary consultation.
- `urgent`: seek veterinary care promptly.
- `invalid`: no triage result due to input/model failure.

Urgency is an ML/product policy layer. It should not be derived only from raw confidence.

## Failure Taxonomy

Use non-200 responses for request/system failures and completed responses for low-confidence model outcomes.

| Code | HTTP | Meaning | UI behavior |
| --- | ---: | --- | --- |
| `invalid_image_type` | 400 | Unsupported MIME type | ask for JPG/PNG |
| `image_too_large` | 413 | File exceeds limit | ask for smaller image |
| `poor_image_quality` | 422 | Blur, darkness, obstruction | ask user to retake image |
| `unsupported_species` | 422 | Species outside scope | block analysis |
| `model_timeout` | 504 | Inference exceeded SLA | offer retry |
| `model_unavailable` | 503 | Service unavailable | retry later |
| `low_confidence` | 200 | Model completed but confidence is below threshold | show uncertainty, ask for better photo or vet consult |
| `out_of_distribution` | 200 | Image is outside training distribution | show validation-pending state |

## Persistence Requirements

Each saved triage should include:

- `triageId`
- `userId`
- `petId`
- image storage reference, not raw base64 in database
- canonical prediction label
- top-k predictions
- confidence and urgency
- OOD score and image quality issues
- model name/version/dataset/calibration versions
- latency and created timestamp
- user feedback fields: `helpful`, `vetConfirmedCondition`, `notes`

## Evaluation Metrics

Minimum ML dashboard:

- upload volume
- completed analysis rate
- timeout rate
- poor-image rate
- OOD rate
- low-confidence rate
- top-1 confidence distribution
- latency p50/p95
- condition distribution
- veterinarian/user feedback agreement
- false-negative review queue for severe/urgent cases

## Safety Requirements

- Never display the result as a diagnosis.
- Always show model version and uncertainty when available.
- Always include a veterinarian escalation path for `consult_soon` and `urgent`.
- Never hide low confidence behind a confident UI treatment.
- Keep recommendations preventive and non-prescriptive.

## Acceptance Criteria

- A mocked service can return this schema and drive the existing result panel.
- The history view can render persisted records without relying on `Math.random()`.
- The UI can distinguish invalid input, timeout, OOD, low confidence, and completed triage.
- Every saved triage records model version and top-k predictions.
- A future evaluation job can compute metrics from persisted records without parsing UI strings.
